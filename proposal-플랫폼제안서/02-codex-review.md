# Codex 리뷰

## 누락된 관점

### 단계별 의사결정 게이트가 없습니다
> "즉시 효과를 내는 **크롬 확장 프로그램** (Phase 1)"
>
> "전사 데이터를 통합하는 **AI 대시보드 플랫폼** (Phase 2)"
>
> "사내 모든 정보를 학습한 **AI 챗봇** (Phase 3)"
>
> "**물류센터까지 확장** 가능한 아키텍처 (Phase 4)"

문서는 4개 Phase를 순차적으로 제시하지만, 각 단계 종료 시점에 무엇이 검증되면 다음 단계로 넘어가는지 없습니다. 경영진 입장에서는 "Phase 1만 먼저 해도 되는가", "어떤 KPI가 충족되지 않으면 다음 단계 투자를 멈추는가"가 핵심입니다.

보완 필요:
- Phase별 성공 기준
- 다음 단계 진입 조건
- 중단 또는 축소 판단 기준

### 운영 주체와 예외 처리 설계가 빠져 있습니다
> "급여 원클릭 자동화 | 급여대장 → 이체서 + 명세서 + 메일 일괄 처리"
>
> "세금계산서 매칭 체크 | 네이버페이/이니시스 결제 ↔ 이카운트 일치 여부 자동 체크"
>
> "직원 누구나 자연어로 질문하면 즉시 답변"

회계, 인사, 채권, 출고 업무는 자동화가 실패하거나 오탐이 나면 즉시 사람이 개입해야 합니다. 현재 초안은 자동화 기능은 많지만, 승인자·검수자·롤백 절차가 없습니다.

보완 필요:
- 업무별 최종 승인자
- 자동 처리 전후 검수 구간
- 실패 시 수동 복구 절차

### 변화관리와 사내 배포 관점이 부족합니다
> "기존 이카운트 ERP 사용 흐름 그대로 유지"
>
> "사이드 패널에서 파일 업로드 → 미리보기 → 실행 3단계"

기존 화면 흐름을 유지하는 것과 실제 현업 정착은 다릅니다. 브라우저 정책, 확장프로그램 배포, 권한 신청, 사용자 교육, 운영 문의 창구가 빠져 있어 도입 난이도가 과소평가되어 보입니다.

보완 필요:
- 사용자군별 교육 계획
- 크롬 확장 배포 방식
- 사내 IT 운영 정책과의 적합성

## 근거 부족한 주장

### ROI 산식이 현재 표와 맞지 않습니다
> "총 투자비 | 8,800만원 + 250만원/월 (운영비)"
>
> "월간 절감 | 367.5만원"
>
> "순 월간 효과 | 117.5만원"
>
> "**투자 회수 기간** | **약 18개월**"
>
> "3년 누적 효과 | **+3,430만원**"

현재 표 숫자대로 계산하면 회수기간은 약 18개월이 아니라 약 75개월입니다. 운영비를 제외하고 계산해도 약 24개월입니다. 3년 누적 효과도 현재 수치와 맞지 않습니다. 이 상태로는 제안서 전체 신뢰도가 크게 떨어집니다.

보완 필요:
- 초기 투자비와 월 운영비를 분리한 재계산
- Phase별 절감 실현 시점 반영
- 3년 누적 효과 산식 공개

### 시간 절감과 오류 감소 수치의 측정 근거가 없습니다
> "월 80시간 | 월 8시간 | **90%**"
>
> "월 15~20건 | 월 1~2건 | **90%**"
>
> "신규 직원 온보딩 | 2주 | 3일 (AI 가이드)"

핵심 설득 포인트가 되는 수치인데, 관측 기간, 표본 업무, 담당자 수, 실제 파일 수가 없습니다. 제안서 외부 독자는 이 수치를 체감이 아니라 측정 결과로 보길 원합니다.

보완 필요:
- 현재 업무 로그 또는 인터뷰 근거
- 업무별 측정 방식
- 샘플 문서 수와 성공률

### "이미 구현된 것"의 검증 수준이 불명확합니다
> "Chrome Extension MV3 전체 구조 (13개 워크플로우 UI)"
>
> "이카운트 Open API 연동 (Zone→Login→CRUD)"
>
> "AI OCR 파이프라인 (Claude API 기반)"

"구현됨", "프로토타입", "고객 데이터로 검증 완료"는 전혀 다른 상태입니다. 현재 문구는 이미 상용 수준으로 준비된 것처럼 읽히는데, 실제 검증 범위가 보이지 않습니다.

보완 필요:
- 데모 가능 범위
- 실제 동작이 확인된 기능과 미완성 기능 구분
- 고객 샘플 데이터 기준 검증 여부

## 검증되지 않은 가정

### "중간 서버 없음" 주장은 AI 처리 구조와 충돌합니다
> "모든 데이터는 이카운트 Open API를 통해 직접 전송, 중간 서버 없음"
>
> "AI Engine — Claude API (OCR/파싱/추천)"

OCR, 파싱, 추천을 Claude API로 처리하면 데이터가 외부 AI API로 이동합니다. 브라우저에서 직접 호출하면 API 키 보관 문제가 생기고, 서버를 두면 "중간 서버 없음" 설명과 충돌합니다. 현재 구조 설명은 보안 질문을 오히려 키웁니다.

보완 필요:
- 데이터 플로우 다이어그램
- API 키 저장 위치
- 프록시 서버 사용 여부

### 일정과 범위가 모두 계획대로 진행된다는 가정이 강합니다
> "Phase 1 | 4주"
>
> "Phase 2 | 8주"
>
> "Phase 3 | 8주"
>
> "Phase 4 | 12주"

13개 워크플로우 자동화, 대시보드, RAG 챗봇, 물류 연동까지 포함한 프로그램치고는 일정이 공격적입니다. 요구사항 변경, 데이터 정제, 외부 API 제약, 현장 피드백 반영이 일정에 거의 반영되지 않은 것으로 보입니다.

보완 필요:
- 선행 조건
- 제외 범위
- 일정 버퍼와 리스크 요인

### 문서 품질이 AI 자동화에 충분하다는 가정이 검증되지 않았습니다
> "PDF 인보이스 → AI OCR → 이카운트 구매입력 자동 변환"
>
> "카카오톡 주문 → AI 파싱 → 이카운트 판매입력 자동 변환"

실제 문서는 촬영 품질, 서식 편차, 누락 값, 오탈자 때문에 성공률 편차가 큽니다. 이 부분이 먼저 검증되지 않으면 Phase 1의 핵심 가치가 무너질 수 있습니다.

보완 필요:
- 샘플 문서 기준 정확도
- 사람이 최종 확인하는 UX
- 오류 케이스 처리 정책

## 대안 미검토

### 실질적인 진입안 비교가 없습니다
> "Phase 1"
>
> "Phase 2"
>
> "Phase 3"
>
> "Phase 4"

형식상 단계형이지만, 읽는 사람에게는 사실상 전체 프로그램 제안으로 보입니다. "Phase 1만 먼저", "Phase 1+2까지만", "전체 도입" 세 가지 옵션을 비교해야 의사결정이 쉬워집니다.

보완 필요:
- MVP안
- 중간안
- 전체안 비교표

### 자체 구축·RPA·BI 도구와의 비교가 없습니다
> "왜 킷에이아이(KiTT AI)인가"

자사 강점은 설명했지만, 고객이 실제로 비교할 대안은 빠져 있습니다. 이카운트 기본 기능 확장, RPA, BI 도구, 타 연동업체와 무엇이 다른지 보여줘야 합니다.

보완 필요:
- 비용 비교
- 구축 기간 비교
- 유지보수 난이도 비교

### AI 챗봇이 꼭 필요한 영역과 아닌 영역 구분이 없습니다
> "CNC코리아의 **모든 사내 정보**를 학습한 AI 챗봇"

모든 문제를 챗봇으로 풀 필요는 없습니다. 조회형 업무는 대시보드나 검색 포털로 충분할 수 있고, 생성형 AI는 일부 고위험 업무에서 제한하는 편이 맞을 수 있습니다.

보완 필요:
- 규칙 기반으로 충분한 영역
- 생성형 AI가 필요한 영역
- 고위험 업무 제한 원칙

## 의사결정 정보 부족

### Phase별 손익분기와 계약 구조가 없습니다
> "총 투자비 | 8,800만원 + 250만원/월 (운영비)"
>
> "Phase 1 시작"

현재는 전체 프로그램 기준 숫자만 있어, 경영진이 "어디까지 먼저 계약할지" 판단하기 어렵습니다. Phase 1 단독 회수기간, Phase 2 추가 투자 시 증분 효과, 유지보수 과금 방식이 필요합니다.

보완 필요:
- Phase별 CAPEX/OPEX
- 검수 후 잔금 구조
- 유지보수 단가와 범위

### 보안·법무·컴플라이언스 질문에 대한 답이 부족합니다
> "급여 정보, 직원 정보 등 개인정보는 암호화 저장"
>
> "계약서·문서"
>
> "직원 누구나 자연어로 질문"

급여, 계약, 거래처 조건, 내부 문서가 모두 AI와 연결되므로, 고객은 기술 스택보다 먼저 데이터 반출 범위와 통제 수준을 봅니다. 현재는 보안 원칙만 있고, 실제 운영 정책이 없습니다.

보완 필요:
- 데이터 분류표
- 외부 전송 대상 데이터 범위
- 권한·로그·삭제 정책

### 수용 테스트 기준이 없습니다
> "핵심 워크플로우 3개 우선 구현 + 현장 테스트"
>
> "샘플 데이터 검증"

"테스트"라는 말만으로는 계약 검수가 어렵습니다. 어느 정도 정확도와 처리 속도를 만족해야 성공인지 없으면, 구축 후 분쟁 포인트가 됩니다.

보완 필요:
- 워크플로우별 성공 기준
- 테스트 샘플 수
- 오류 허용치

### 고객 현황 기초 데이터가 빠져 있습니다
> "월 60~80시간의 비효율"
>
> "다수의 반복 업무"

사용자 수, 월 주문 건수, 문서량, 물류량이 없어 현재 범위가 과한지 부족한지 판단하기 어렵습니다. 제안 범위와 가격이 고객 실정에 맞는지 확인할 기초 데이터가 없습니다.

보완 필요:
- 현행 프로세스 맵
- 사용자 수와 부서별 사용 방식
- 월간 주문·문서·물류 처리량

## 총평

현재 초안은 "무엇을 만들 수 있는가"는 잘 보여주지만, "왜 지금 이 범위로 투자해도 되는가"를 뒷받침하는 수치와 검증 구조가 약합니다. 우선순위는 `ROI 재계산`, `Phase 1 단독안 제시`, `보안/승인 프로세스 명시` 순으로 보강하는 것이 맞습니다.
