## cmux 워크스페이스 레이아웃

```
┌─────────────────┬──────────────────┬──────────────────┐
│  cmux Browser   │  Claude (작성자)  │  Codex (리뷰어)   │
│                 │                  │                  │
│  도입제안서 HTML │  $ claude        │  $ codex         │
│  (원본 참조)     │                  │                  │
└─────────────────┴──────────────────┴──────────────────┘
```

- 워크스페이스 디렉토리: `analysis-도입제안서/`
- 좌측 pane: cmux 빌트인 브라우저로 `file://`로 HTML 렌더링
- 가운데/오른쪽 pane: 각각 `claude`, `codex` CLI 실행
- cmux는 GUI 앱이므로 셋업 스크립트가 워크스페이스를 자동 생성하지는 않음 — 사용자가 직접 cmux에서 워크스페이스를 열고 split

## README.md 협업 프로토콜

두 에이전트가 시작 시 반드시 읽는 공통 규약. 다음을 포함:

### 분석 차원 (6개)

1. **무엇을(What)**: 제안의 핵심 — 무엇을 도입하자는 것인가
2. **왜(Why)**: 도입 동기, 현재 문제, 기대 효과
3. **어떻게(How)**: 도입 방식, 통합 지점, 운영 모델
4. **리스크**: 기술적·운영적·보안·법적 리스크
5. **비용·ROI**: 도입 비용, 운영 비용, 회수 시점
6. **권고**: Go / No-go / 조건부 Go (조건 명시)

### 작성 규칙 (작성자 = Claude)

- 언어: 한국어, 마크다운
- 근거 명시: 제안서 내 근거는 인용, 추정은 "(추정)" 명시
- 톤: 객관·중립. 옹호·폄하 없음
- 빠진 정보는 "정보 부족" 섹션에 명시

### 리뷰어 체크리스트 (Codex)

리뷰는 다음 5축으로 진행:

- [ ] **누락된 관점**: 작성자가 빠뜨린 분석 차원/이해관계자
- [ ] **근거 부족한 주장**: 제안서에 없거나 출처 불명확한 단정
- [ ] **검증되지 않은 가정**: "당연히 ~할 것이다" 식의 암묵 가정
- [ ] **대안 미검토**: 비교했어야 할 대안(자체 구축, 타사, 미도입 등)
- [ ] **의사결정 정보 부족**: 결정에 필요한데 빠진 데이터·질문

각 항목은 `01-claude-draft.md`의 어느 부분에 대한 것인지 인용으로 명시.

### 수정 규칙 (작성자 2차)

리뷰 항목별로 다음 마커로 응답:

- ✅ **반영**: 어떻게 반영했는지 한 줄 설명
- ❌ **반박**: 반영하지 않은 이유
- ⚠️ **부분 반영**: 일부만 수용한 이유와 범위

## 셋업 작업 (이 설계 승인 후)
